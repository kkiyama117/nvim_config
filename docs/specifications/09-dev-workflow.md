# 09 — Developer workflow spec

Defines the **cross-cutting** normative rules for code-generation scripts in this repository (`nvim_config`) — the rules that apply to *every* generator regardless of what it generates. Referenced from [`AGENTS.md`](../../AGENTS.md) §3 and from individual designs under [`specifications/implementation/`](implementation/).

All AI agents and human maintainers follow this spec. The first consumer is [deps-docs-auto-gen](implementation/2026-07-13-deps-docs-auto-gen-design.md).

## 1. Purpose

- Single canonical home for the rules that genuinely cross-cut multiple generators.
- Keep generated artifacts byte-for-byte reproducible so the pre-commit hook can enforce freshness via `git diff --exit-code`.
- Avoid introducing a second copy of truth for any data that is already derivable from a source file.

## 2. Scope

In scope:

- The 6 cross-cutting rules in §3 (R1, R6, R7, R8, R9, H5) that apply to every generator.
- The tasks registry (§4) and pre-commit hook installation mechanism (§5) shared by all generators.

Out of scope (deferred to per-generator designs):

- Per-generator choices: runtime language, dependency source, file paths, classification logic, exit-code table, phase split. These live in each generator's design under [`implementation/`](implementation/) as invariants (I1, I2, …) and implementation detail (§5, §6, …).
- When a second generator is added, this spec may grow to absorb rules that prove genuinely cross-cutting. Until then, per-generator designs carry the specifics.
- Runtime nvim behavior (covered by `00-document-management.md` and per-plugin designs).
- CI runner configuration (no CI is set up yet; the pre-commit hook is the only enforcement surface).

## 3. Normative rules

The 6 rules below are the only normative rules at this spec level. They are cross-cutting: any generator added to this repo must satisfy them. Per-generator rules (idempotency of a specific output, specific exit codes, specific file paths beyond the `scripts/` root, etc.) live in each generator's design doc as invariants.

| ID | Rule |
|----|------|
| R1 | Every generator lives under `scripts/`. No generator may live at the repo root, under `lua/`, or under `deps/`. (Cross-cutting: applies to every future generator regardless of what it generates.) |
| R6 | Generators are **idempotent**: running twice produces byte-identical output. Ordering is deterministic (source-file order → insertion order → sorted only where the rendered format requires it). |
| R7 | Generators never read their own prior output as a *source of data*. The source files (e.g. `deps/*.toml`) are the only input. Reads to locate sentinel markers (for in-place block replacement) are permitted — they do not feed plugin data into the model. |
| R8 | Generated files begin with a marker header: `<!-- AUTO-GENERATED by scripts/<gen>.ts — do not edit -->` for markdown, an equivalent header comment for other formats. The "generated" signal is carried by this header, not by the filename. |
| R9 | Generators exit non-zero with a documented exit code on every failure mode. Exit codes are stable and listed in the generator's design doc. |
| H5 | A generator whose source files are unchanged must produce zero diff. The pre-commit hook is therefore a no-op for commits that don't touch generator inputs. (Consequence of R6 + R7: idempotent output + no reading of prior output → unchanged inputs ⇒ unchanged output.) |

**Removed (pass-1 review, 2026-07-13):** G3 (in-body `Last updated: YYYY-MM-DD` line) was removed — it violated H5 (a date derived from the system clock changes even when inputs are unchanged). Generated-file freshness dates are available via `git log <file>`; no in-body date line is required. See [deps-docs-auto-gen design §8](implementation/2026-07-13-deps-docs-auto-gen-design.md) A3/D8 decision.

**Removed (pass-2 review, 2026-07-13):** H4 (no network in hook) was removed — pass-1 D7 dropped `--no-remote`, so the hook may fetch from JSR on first run (deno's default HTTP cache). The hook's network footprint is limited to deno's default HTTP cache (first run only). See [deps-docs-auto-gen design §8](implementation/2026-07-13-deps-docs-auto-gen-design.md) D7 decision and I7.

**Re-elevated (pass-2 review, 2026-07-13):** R1 (generators live under `scripts/`) was re-elevated from the deferred list — pass-2 D-pass2-2 found it is genuinely cross-cutting (every future generator must live under `scripts/`), not per-generator, and was incorrectly listed as deferred in pass-1.

**Deferred to per-generator designs:** R2 (Deno/TS only), R3 (`deno task` entrypoint), R4 (JSR via `deno.json`), R5 (cache strategy), R10 (sentinel or whole-file rewrite), G1 (git-tracked), G2 (no hand-edits), G4 (pure-data Lua module), H1 (symlinked hook), H2 (hook runs every gen task), H3 (no auto-stage). These are carried across the deps-docs-auto-gen design's invariants (I1–I8), success criteria (S2), and body text (§2 alternatives, §5 implementation detail, §6 hook) — see the mapping in [deps-docs-auto-gen design §8a](implementation/2026-07-13-deps-docs-auto-gen-design.md). They will be re-elevated into this spec only when a second generator proves them genuinely cross-cutting.

## 4. Tasks registry

The canonical list of `deno task` names exposed by `scripts/deno.json`. New generators append a row here.

| Task | Purpose | First introduced by |
|------|---------|---------------------|
| `gen` | Regenerate all deps-docs artifacts (`deps/README.md` sentinel block, `lua/dpp_min_deps.lua`, `docs/references/deps-list.md`, `docs/README.md` index line). Entry: `deno run --allow-read=. --allow-write=<outputs> scripts/gen_deps.ts`. | deps-docs-auto-gen |

When a second generator is added, its task name should be `gen-<topic>` and the umbrella `gen` task should chain all `gen-*` tasks (so `deno task gen` remains the "regenerate everything" entrypoint).

Note (per pass-1 D7 decision): the `gen` task does **not** use `--no-remote`. Deno's default HTTP cache is relied upon: the first run after a fresh clone may fetch `@std/toml` from JSR (network required once), and subsequent runs are served from the local cache. This eliminates the "module not found in cache" UX hazard on fresh clones. An optional `deno cache scripts/gen_deps.ts` can be run to pre-populate the cache if strict offline-first run is desired.

## 5. Pre-commit hook installation

Tracked script: `scripts/install-hooks.sh`. Run once after clone:

```sh
sh scripts/install-hooks.sh
```

The script symlinks `.git/hooks/pre-commit` → `scripts/hooks/pre-commit` (tracked), so future edits to the hook flow without re-running the installer. The installer is idempotent and safe to re-run.

The hook itself (`scripts/hooks/pre-commit`) is the canonical implementation; the deps-docs-auto-gen design §6 shows its initial content. The hook runs `deno task gen` then `git diff --exit-code` on the generated files; it does not auto-stage (the user reviews and stages the diff themselves).

## 6. Fresh-clone onboarding

A new contributor (or a fresh checkout) does:

1. `sh scripts/install-hooks.sh` — install the pre-commit hook (one-time).
2. `deno task gen` — verify generated files are up to date (should be a no-op if the repo is clean; first run may fetch `@std/toml` from JSR).

After that, the pre-commit hook keeps generated files fresh automatically. (The previous `deno task cache` step is no longer required — deno's default HTTP cache handles it transparently.)

## 7. Revision procedure

This spec is normative. When revising:

1. Raise the revision reason as an issue in `docs/issues/` (per [`00-document-management.md`](00-document-management.md) §10).
2. If the change is non-trivial, go through a design under `specifications/implementation/`.
3. Edit this file directly and include the issue path in the commit log.
4. Sync affected designs and `AGENTS.md` in the same commit.

## 8. Relationship to other specs

- [`00-document-management.md`](00-document-management.md) — placement/naming/lifecycle of all docs. This spec inherits its vocabulary and lifecycle.
- [`AGENTS.md`](../../AGENTS.md) §2 — hard constraints. A new generator that would violate a hard constraint must be approved per the AGENTS.md escalation rule.
- Per-implementation designs under [`implementation/`](implementation/) — each generator's design doc carries its own success criteria (S1–S7), invariants (I1–I8), and exit-code table. This spec is the slim cross-cutting baseline (6 rules: R1, R6, R7, R8, R9, H5); designs may tighten but not loosen the rules here. When a per-generator invariant proves genuinely cross-cutting (verified by a second generator), it is elevated into this spec via the revision procedure (§7).
