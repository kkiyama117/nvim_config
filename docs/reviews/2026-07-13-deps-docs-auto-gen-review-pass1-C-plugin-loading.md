## Verified premises (no action needed)

- **C-2 (repo-string consumption): S7's strict sub-claim holds.** `dpp_loader.lua` consumes ONLY repo strings from `minimum_deps` / `normal_deps`. Evidence: the two lists are arrays of bare strings (lines 23-36). `load_plugins(list_of_plugin)` (lines 70-78) iterates the list and uses each element solely as a plugin-name string for `dest_path(name)` (line 72), `is_plugin_ready(name, ...)` (line 73), `install_github_plugin(name, ...)` (line 74), and `vim.opt.runtimepath:prepend(vim.fs.normalize(plugin_dest_path))` (line 76). No `rtp` / `depends` / `on_*` / `hook_*` field is read from these lists. The lists are NOT passed to `dpp.load_state` (line 97) or `dpp.make_state` (lines 107, 119) â€” those take `(dpp_cache_home, dpp_denops_script)`. `dpp#make_state`'s actual input is the `ConfigReturn` built by `denops/dpp.ts`'s `Config.config()` from the TOML files (`denops/dpp.ts` line 147: `Deno.readDirSync(dppTomlDir)` â†’ `tomlExt.actions.load.callback`), which the refactor does not touch. Therefore a generated module carrying only repo strings is exactly the right shape, and `dpp#make_state`'s input is byte-identical before and after. (The HIGH finding C2 above is about the *bootstrap* behavior change via `load_plugins`, not about `dpp#make_state` input.)
- **C-1: minimum/normal split is correct.** `minimum_deps = {Shougo/dpp.vim, Shougo/dpp-ext-lazy}` is exactly what `dpp_loader.lua` needs before `dpp.load_state` (line 97): both have `rtp = ''` in `deps/dpp.toml` (lines 7, 11) and are rtp-prepended by `load_plugins(minimum_deps)` (line 92) so `require("dpp")` (line 94) and `dpp.load_state` resolve. `normal_deps = rest of dpp.toml + vim-denops/denops.vim` is what `load_plugins(normal_deps)` (line 99) clones + rtp-prepends so `dpp-ext-toml`, `dpp-ext-installer`, `dpp-ext-lazy`, `dpp-protocol-*`, and `denops.vim` are available before `dpp.make_state` (line 107) and `ensure_denops_plugin()` (line 100). The Â§3.2 predicate matches the current hand-maintained intent.
- **C-3: `rtp = ''` is orthogonal to classification.** The Â§3.2 / Â§5.4 predicate uses only `repo âˆˆ MINIMUM_REPOS`; `rtp` is never consulted by the script. `rtp = ''` on `dpp.vim` and `dpp-ext-lazy` (dpp.toml lines 7, 11) is a dpp-ext-toml directive telling dpp NOT to add them to rtp when loading from TOML (because they're already on rtp via the bootstrap loader at dpp_loader.lua line 76). All `dpp-ext-*` and `dpp-protocol-*` plugins in `deps/dpp.toml` are correctly classified: `dpp-ext-lazy` â†’ minimum; `dpp-ext-toml`, `dpp-ext-local`, `dpp-ext-installer`, `dpp-ext-packspec`, `dpp-protocol-git`, `dpp-protocol-http` â†’ normal.
- **C-6: no autocmd interference with the sentinel rewrite.** The `dpp-ext-toml` `hook_add` autocmd `CursorHold */rc/*.toml call dpp#ext#toml#syntax()` (`deps/dpp.toml` line 18) is (a) guarded by `if !has('nvim')` (line 17) so it only fires on Vim, not Neovim, and (b) the glob is `*/rc/*.toml`, NOT `deps/*.toml`, so it does not match `deps/README.md` or `deps/*.toml` in any case. No `after/` or `ftplugin/` directory exists at the repo root (the only ftplugin content is `lua/dpp_hooks/ft.lua`, referenced by `deps/merge.toml`'s non-dpp `hook_files` key and consumed via dpp's `hooksFile` mechanism at `makeState` time, not at sentinel-rewrite time). The sentinel block rewrite is a one-shot operation by the generator script, outside Neovim, so it cannot interfere with any startup autocmd.
- **C-7: the Â§5.4 "classified twice" check is reachable and correct.** Scenario: a future maintainer adds `[[plugins]] repo = 'vim-denops/denops.vim'` to `deps/dpp.toml` (e.g. for eager load). - **I4 / I5 (startup order): confirmed.** The generated module is pure data (`return { minimum_deps = {...}, normal_deps = {...} }`), required at the top of `dpp_loader.lua` before `initialize_dpp()` runs (line 147). `vim.loader` is enabled in `init.lua` line 5, so the `require` resolves synchronously from bytecode cache (or compiles on first require).
