# Review: deps-docs-auto-gen — Reviewer-A Architecture

Reviewed: docs/specifications/implementation/2026-07-13-deps-docs-auto-gen-design.md
Reviewer: composer-2.5-fast (Reviewer-A)
Date: 2026-07-13
Status: in-review

## Verdict

**Block on A1/A2/A3; approve with revisions on the rest.** The overall shape (single TOML source of truth → pure-data generated Lua module → `require()` in `dpp_loader.lua` → pre-commit freshness hook) is architecturally sound and the module boundary between `scripts/` and `lua/` is clean. The placement of `09-dev-workflow.md` under `docs/specifications/` (normative) is correct.

However three findings block approval. **A1 (CRITICAL)**: the design pairs `require("dpp_min_deps.generated")` with a file at `lua/dpp_min_deps.generated.lua`, but Neovim treats `.` as a directory separator (`:help lua`), so the require resolves to `lua/dpp_min_deps/generated.lua` and will fail — aborting `init.lua:41` `require('dpp_loader')` at startup. **A2 (HIGH, cross: C)**: the §5.5 generated `normal_deps` adds `Shougo/dpp-protocol-http` (present in `deps/dpp.toml:35`, absent from the current hand-maintained `dpp_loader.lua:29-36`) and reorders entries, so S7 ("byte-identical") is violated by the design's own example and the Phase 2 acceptance ("prints the same two lists") is unmettable. **A3 (HIGH, cross: D)**: the `Last updated: YYYY-MM-DD` line derived from the system clock breaks I2/H5 when `deps/*.toml` is unchanged but a day rolls over — the hook would then block unrelated commits.

## Findings

| ID | Severity | Status | Location | Note |
|----|----------|--------|----------|------|
| A1 | CRITICAL | open | §5.5, §3.1, §4 Phase 2, I4/I5, S2; `09-dev-workflow.md` §4 | Module name / file path mismatch breaks `require()` resolution. Neovim treats any `.` in a require path as a directory separator (`:help lua`: "For a module `foo.bar`, each directory is searched for `lua/foo/bar.lua`, then `lua/foo/bar/init.lua`"). `require("dpp_min_deps.generated")` therefore resolves to `lua/dpp_min_deps/generated.lua`, **not** `lua/dpp_min_deps.generated.lua` as the design consistently places it (S2, §3.1 diagram, §5.1, §5.5, §6 hook diff list, `09-dev-workflow.md` §4 task row). Since `init.lua:41` calls `require('dpp_loader')` unprotected and the refactored `dpp_loader.lua` calls `require("dpp_min_deps.generated")` at its top (§4 Phase 2), the error propagates and aborts init before `filetype indent plugin on` / `syntax on` (`init.lua:43-44`). I5 ("startup order is unchanged") is false as written. Fix: either (a) move the file to `lua/dpp_min_deps/generated.lua` and keep the require as-is, or (b) rename the module to `dpp_min_deps_generated` (underscore) with file `lua/dpp_min_deps_generated.lua`. Update S2, S5, §3.1, §5.1, §5.5, §6, and `09-dev-workflow.md` §4 consistently. |
| A2 | HIGH | open | §5.5, §4 Phase 2 acceptance, S7 (cross: C) | Generated `normal_deps` differs from the current hand-maintained list — the exact drift the issue was opened for, but the design assumes it away. `deps/dpp.toml:35` contains `Shougo/dpp-protocol-http`, but `dpp_loader.lua:29-36` `normal_deps` = {installer, local, packspec, toml, protocol-git, denops.vim} omits it. The §5.5 generated `normal_deps` includes `dpp-protocol-http` and uses TOML order (toml, local, installer, packspec, protocol-git, protocol-http, denops.vim), differing in both content and order. `load_plugins()` (`dpp_loader.lua:70-78`) prepends each entry to `runtimepath`, so content + order flow into `dpp#make_state` input → S7 ("byte-identical") is violated by the design's own example, and the Phase 2 acceptance ("prints the same two lists as the previous hand-maintained tables") cannot pass. Fix: either (i) explicitly acknowledge the drift and reframe Phase 2 acceptance as "generated list is the *corrected* superset; runtime impact of newly eagerly-loading `dpp-protocol-http` verified safe (Reviewer-C sign-off)", or (ii) land a prior commit adding `dpp-protocol-http` to the hand-maintained list so the generator is purely mechanical and S7 holds. |
| A3 | HIGH | open | §5.7, I2, `09-dev-workflow.md` H5 (cross: D) | `Last updated: YYYY-MM-DD` derived from `new Date().toISOString().slice(0,10)` breaks idempotency. If `deps/*.toml` is unchanged but the script runs on a later day (e.g. an unrelated commit triggers the pre-commit hook the day after the last regen), the date line changes → `git diff --exit-code` is non-zero → the hook blocks an unrelated commit. This directly contradicts I2 ("running twice produces byte-identical output") and `09-dev-workflow.md` H5 ("A generator whose source files are unchanged must produce zero diff"). The design notes the date "changes whenever `deps/*.toml` changes" but does not address the unchanged-but-day-rolled case. Fix: (a) derive the date from the max mtime of the source TOMLs, (b) drop the in-body date line and rely on git history, or (c) keep the existing date line when the non-date content is unchanged (read prior line, preserve on content match). |
| A4 | MEDIUM | open | §3.3 I1 vs §5.6 | I1 ("the script never reads `lua/dpp_loader.lua`, `deps/README.md`, or any prior generated output") is contradicted by the §5.6 sentinel-rewrite mechanism, which "reads `deps/README.md`, asserts both sentinel lines exist, and replaces the content between". A marker-preserving in-place rewrite *requires* reading the existing file to locate the sentinels and preserve surrounding content. I1 as stated is therefore not an actual invariant; a reviewer enforcing it literally would reject the sentinel approach. Fix: scope I1 to "never reads `deps/README.md` (or any generated output) as a *source of plugin data*; reads to locate sentinel markers only are permitted". |
| A5 | MEDIUM | open | §4 | "Each phase is independently revertable" is too strong. The Phase 4 hook runs `deno task gen` then `git diff --exit-code` on the generated files; reverting Phase 2 or 3 while keeping Phase 4 makes `gen` either fail (no `render_lua.ts`) or regenerate now-untracked files → `git diff` non-zero → every commit is blocked. Reverting Phase 2 alone also leaves `dpp_loader.lua` requiring a deleted module → nvim startup aborts (cf. A1). Phases are *sequentially* revertable (revert N..1 in order), not independent. Fix: state the ordering explicitly ("to roll back, revert Phase 4 first, then 3 → 2 → 1") or note that Phase 4 must be disabled before any earlier phase is reverted. |
| A6 | LOW | open | §5.5, §4 Phase 2, `09-dev-workflow.md` G4 | The `minimum_deps` / `normal_deps` field-name contract between `render_lua.ts` (producer) and `dpp_loader.lua` (consumer) is implicit and unenforced across the Lua/TS boundary. G4 only pins "pure data: only `return { ... }`", not the field names. If either side renames, `dpp_loader.lua` silently gets `nil` and `load_plugins(nil)` errors at startup. Fix: add a one-line guard in `dpp_loader.lua` after the require (`assert(deps and deps.minimum_deps and deps.normal_deps, "dpp_min_deps.generated missing fields")`), or pin the field names in `09-dev-workflow.md` G4 / a comment in `render_lua.ts`. |
| A7 | LOW | open | §3.2, §5.4, §5.5 | The §3.2 claim that the minimum set is "encoded **once** … the only place where the minimum set is named explicitly" is true in *code* (the `MINIMUM_REPOS` constant in §5.4) but the set `{Shougo/dpp.vim, Shougo/dpp-ext-lazy}` is restated in design prose at §1.1, §3.2, §5.4, and §5.5. If the set changes, all four prose sites must update or the design becomes self-inconsistent. Fix: scope the claim to "in code" and add a one-line note that prose restatements are illustrative. |

## Verified premises (no action needed)

- **A-5 placement (09-dev-workflow.md)**: `docs/specifications/` is the correct home. Per `00-document-management.md` §3, `spec` = "Project-wide normative spec" at `specifications/NN-<topic>.md`; `docs/references/` is for "External / host-state reference material". 09-dev-workflow.md contains normative rules (R1–R10, G1–G4, H1–H5) governing all generators — it is not external/host-state reference material. The single-current-generator situation does not change its normative character (specs are forward-looking baselines by design). (Whether it is *premature* is D-4's YAGNI angle, not a placement question.)
- **A-1 mechanism (modulo the A1 dot-resolution bug)**: the *mechanism* of a pure-data `require()`d module preserving startup order is sound. `vim.loader.enable()` runs at `init.lua:4-6` before `require('dpp_loader')` at `init.lua:41`, so the generated module is byte-code cached and resolves synchronously with no new autocmds / `User` events. I4 (pure data, no `vim.*`, no side effects, no nested `require`) holds for the §5.5 shape. Only the dot-resolution defect (A1) breaks this.
- **A-2 module boundary**: the `scripts/` (generator) ↔ `lua/` (consumer) boundary is clean. The generator never reads `dpp_loader.lua`; the consumer never references `scripts/`. The sole coupling is the module name + field-name contract (see A6).
- **I3 / I6 / I8**: hold as stated. Generated marker headers are present in §5.5/§5.7; the §6 hook script does not `git add` (only `git diff --exit-code` + exit 1), matching I6 and `09-dev-workflow.md` H3; a missing sentinel yields exit code 3 per §5.9 / §5.8.
- **I7**: holds *given* the documented `deno task cache` precondition (`--no-remote --no-npm` + pre-populated JSR cache). The invariant is conditional; the precondition is documented in `09-dev-workflow.md` §6. (The fresh-clone UX hazard of that precondition is D-7's scope.)

## Open questions back to author

- **A1**: which fix do you prefer — `lua/dpp_min_deps/generated.lua` (keep the dotted require, add a one-file directory) or `lua/dpp_min_deps_generated.lua` + `require("dpp_min_deps_generated")` (underscore, no new directory)? The latter is one fewer moving part. Whichever is chosen, S2/S5, §3.1, §5.1, §5.5, §6, and `09-dev-workflow.md` §4 must be updated in lockstep.
- **A2**: was `dpp-protocol-http` intentionally omitted from the current hand-maintained `normal_deps`, or is it silent drift? If drift, should the design explicitly call out the runtime behavior change (HTTP protocol now eagerly bootstrap-loaded) and require Reviewer-C sign-off on safety, or should a prior commit add it to the hand-maintained list so the generator is purely mechanical and S7 holds?
- **A3**: is the in-body `Last updated:` line worth the idempotency cost? If yes, which deterministic source do you prefer — max mtime of the source TOMLs, or "keep existing date when non-date content is unchanged"?
- **A6**: should the field-name contract be pinned in `09-dev-workflow.md` G4, enforced at runtime via an `assert` in `dpp_loader.lua`, or both?
